# 错误提示优化指南

## 📋 优化内容

已实现用户友好的错误提示系统，将技术性错误转换为易懂的提示信息。

---

## ✅ 已优化内容

### 1. 创建错误映射工具类

**文件**: `lib/core/utils/error_message_mapper.dart`

**功能**:
- 将 `DioException` 转换为用户友好提示
- 过滤技术性错误关键词（如 "bad response", "type 'xxx' is not a subtype" 等）
- 根据 HTTP 状态码返回具体提示
- 统一错误消息格式

**常见错误映射**:
```
技术错误                    →  用户友好提示
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
bad response               →  服务器错误，请稍后重试
type 'String' is not...    →  操作失败，请稍后重试
SocketException            →  网络连接失败，请检查网络设置
Failed host lookup         →  无法连接到服务器，请检查网络
Connection timeout         →  连接超时，请检查网络后重试
401 Unauthorized          →  登录已失效，请重新登录
404 Not Found             →  请求的内容不存在
500 Server Error          →  服务器错误，请稍后重试
```

### 2. 优化 API 拦截器

**文件**: `lib/core/network/api_interceptor.dart`

**改进**:
- 使用 `ErrorMessageMapper` 统一处理错误
- 自动过滤技术性错误
- 保留后端返回的业务错误信息

**优化前**:
```dart
// ❌ 硬编码错误提示，无法过滤技术错误
String errorMessage = '哎呦,出错啦~~';
switch (err.type) {
  case DioExceptionType.connectionTimeout:
    errorMessage = '网络超时,请稍后重试';
    break;
  // ... 大量 case 代码
}
```

**优化后**:
```dart
// ✅ 统一使用工具类处理
final userFriendlyMessage = ErrorMessageMapper.mapDioException(err);
```

## 📖 使用指南

### Provider 层错误处理标准模式

#### 推荐模式（已在 AuthProvider 中使用）

```dart
Future<void> someAction() async {
  try {
    LoadingHUD.show('处理中...');
    
    // 调用 Service
    await _service.doSomething();
    
    LoadingHUD.dismiss();
    ToastUtil.success('操作成功');
  } on DioException catch (e) {
    // ✅ DioException 已被拦截器处理，直接使用 error 字段
    LoadingHUD.dismiss();
    final errorMsg = e.error?.toString() ?? '操作失败';
    ToastUtil.error(errorMsg);
    rethrow;
  } on Exception catch (e) {
    // ✅ Service 层抛出的业务异常
    LoadingHUD.dismiss();
    final errorMsg = ErrorMessageMapper.mapException(e);
    ToastUtil.error(errorMsg);
    rethrow;
  } catch (e) {
    // ✅ 兜底：未预期错误
    LoadingHUD.dismiss();
    ToastUtil.error('操作失败，请稍后重试');
    print('❌ [未预期错误] $e');
    rethrow;
  }
}
```

### Service 层错误处理标准模式

```dart
Future<MyModel> fetchData() async {
  try {
    final response = await _dioClient.get('/api/data');
    
    // ✅ 检查业务状态码
    if (response.data['code'] != 100000) {
      final msg = response.data['msg'];
      throw Exception(msg is List ? msg.join(', ') : (msg ?? '获取数据失败'));
    }
    
    // ✅ 检查数据是否为空
    final data = response.data['data'];
    if (data == null) {
      throw Exception('返回数据为空');
    }
    
    return MyModel.fromJson(data);
  } on DioException catch (e) {
    // ✅ DioException 交给拦截器处理，直接向上抛出
    rethrow;
  } catch (e) {
    // ✅ 其他异常（如解析错误）转换为 Exception
    throw Exception('数据处理失败: $e');
  }
}
```

## 🎯 错误提示优先级

### 优先级顺序

1. **后端业务错误** (最高优先级)
   - 来自 `response.data['msg']`
   - 示例: "手机号格式错误", "验证码已过期"
   - ✅ 直接显示给用户

2. **HTTP 状态码错误**
   - 401, 404, 500 等
   - 通过 `ErrorMessageMapper` 转换
   - ✅ 显示友好提示

3. **网络错误**
   - 超时、连接失败等
   - 通过 `ErrorMessageMapper` 转换
   - ✅ 显示网络相关提示

4. **兜底提示** (最低优先级)
   - 无法识别的错误
   - 默认: "哎呦，出错啦~~"

### 实际场景示例

```dart
// 场景1: 后端返回业务错误
response: {code: 100001, msg: "手机号格式错误"}
→ 显示: "手机号格式错误"

// 场景2: 网络超时
DioExceptionType.connectionTimeout
→ 显示: "连接超时，请检查网络后重试"

// 场景3: 401 未授权
statusCode: 401
→ 显示: "登录已失效，请重新登录"
→ 自动跳转登录页

// 场景4: 技术错误 (类型转换失败)
type 'String' is not a subtype of type 'int'
→ 显示: "操作失败，请稍后重试"  ← 过滤技术细节
```

## 📌 注意事项

### ✅ DO - 应该做的

1. **在 Provider 中捕获异常时**:
   - 优先捕获 `DioException`（使用 `e.error` 获取友好提示）
   - 其次捕获 `Exception`（使用 `ErrorMessageMapper.mapException(e)`）
   - 最后兜底 `catch (e)`（显示默认错误提示）

2. **在 Service 中抛出异常时**:
   - 抛出有意义的 `Exception('具体错误描述')`
   - DioException 直接 `rethrow`，交给拦截器处理

3. **显示错误提示时**:
   - 使用 `ToastUtil.error()` 显示错误
   - 使用 `LoadingHUD.dismiss()` 关闭加载框

### ❌ DON'T - 不应该做的

1. **不要硬编码错误提示**:
   ```dart
   // ❌ 错误
   catch (e) {
     ToastUtil.error('网络请求失败');
   }
   
   // ✅ 正确
   on DioException catch (e) {
     final errorMsg = e.error?.toString() ?? '操作失败';
     ToastUtil.error(errorMsg);
   }
   ```

2. **不要吞掉异常**:
   ```dart
   // ❌ 错误 - 异常被吞掉，无法向上传递
   catch (e) {
     ToastUtil.error('失败');
     // 没有 rethrow!
   }
   
   // ✅ 正确
   catch (e) {
     ToastUtil.error('失败');
     rethrow;  // ← 向上传递异常
   }
   ```

3. **不要重复显示错误提示**:
   ```dart
   // ❌ 错误 - Service 和 Provider 都显示 Toast
   // Service:
   catch (e) {
     ToastUtil.error('失败');  // ← 不应该在这里显示
     rethrow;
   }
   
   // Provider:
   catch (e) {
     ToastUtil.error('失败');  // ← 会重复显示
   }
   
   // ✅ 正确 - 只在 Provider 显示
   // Service:
   catch (e) {
     throw Exception('失败原因');  // ← 只抛出异常
   }
   
   // Provider:
   on Exception catch (e) {
     ToastUtil.error(ErrorMessageMapper.mapException(e));  // ← 只在这里显示
   }
   ```

## 🔄 迁移指南

### 现有代码需要修改的地方

#### 1. Provider 中的错误处理

查找所有 `on Exception catch (e)` 代码:


**修改前**:
```dart
on Exception catch (e) {
  LoadingHUD.dismiss();

  ToastUtil.error(errorMsg);
  rethrow;
}
```

**修改后**:
```dart
on DioException catch (e) {
  // ✅ DioException 已被拦截器处理
  LoadingHUD.dismiss();
  final errorMsg = e.error?.toString() ?? '操作失败';
  ToastUtil.error(errorMsg);
  rethrow;
} on Exception catch (e) {
  // ✅ 使用 ErrorMessageMapper
  LoadingHUD.dismiss();
  final errorMsg = ErrorMessageMapper.mapException(e);
  ToastUtil.error(errorMsg);
  rethrow;
}
```

#### 2. Service 中的错误处理

保持现有逻辑，只需确保:
- DioException 直接 `rethrow`
- 业务错误抛出清晰的 `Exception('具体原因')`

## ✨ 总结

### 核心优化

1. **创建统一的错误映射工具** (`ErrorMessageMapper`)
2. **优化 API 拦截器**，自动过滤技术错误
3. **标准化 Provider 错误处理模式**
4. **保留后端业务错误信息**，提升用户体验

### 用户体验提升

- ✅ 不再显示 "bad response"、"type 'xxx' is not..." 等技术错误
- ✅ 网络错误提示更清晰（超时、连接失败、DNS 错误等）
- ✅ HTTP 状态码错误有具体说明
- ✅ 后端业务错误准确传达给用户
- ✅ 兜底提示更友好（"哎呦，出错啦~~"）
