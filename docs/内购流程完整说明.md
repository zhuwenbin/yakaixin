## 🔍 关键机制详解

### 1. 清理模式 (Clearing Mode)

#### 什么是清理模式？

```dart
/// 清理所有待处理的交易
/// 作用：避免 storekit_duplicate_product_object 错误
Future<void> _clearPendingTransactions() async {
  try {
    debugPrint('清理模式：开始');

    _isCleaningMode = true;
    _cleaningCompleter = Completer<void>();

    // 触发恢复购买
    await _iap.restorePurchases();

    try {
      // ⏰ 3秒超时！
      await _cleaningCompleter!.future.timeout(const Duration(seconds: 3));
    } catch (e) {
      debugPrint('清理超时，可能没有待处理交易');
    }

    _isCleaningMode = false;
    debugPrint('清理模式：完成');
  } catch (e) {
    _isCleaningMode = false;
  }
}
```

#### 为什么需要清理模式？

**场景 1: 用户支付成功，但应用崩溃**
```
用户点击购买
    ↓
Face ID 验证
    ↓
支付成功 ✅
    ↓
应用正在验证...
    ↓
❌ 应用崩溃/被杀死
    ↓
交易未完成 (pending)
    ↓
下次启动:
    ↓
清理模式检测到 pending transaction
    ↓
保存收据到缓存
    ↓
完成交易 (避免重复扣款)
    ↓
自动补发验证
    ↓
权限开通 ✅
```

**场景 2: 网络错误导致验证失败**
```
支付成功 ✅
    ↓
开始验证收据
    ↓
❌ 网络超时/服务器错误
    ↓
收据已保存到缓存
    ↓
完成交易 (避免重复扣款)
    ↓
下次启动:
    ↓
自动补发验证
    ↓
权限开通 ✅
```

#### 清理模式的特点

| 特性 | 说明 | 为什么这样设计 |
|------|------|---------------|
| **3秒超时** | 等待 3 秒后自动结束 | 避免启动卡死，快速进入应用 |
| **不验证收据** | 只保存收据，不立即验证 | 快速完成交易，避免启动延迟 |
| **保存到缓存** | 收据保存到 SharedPreferences | 下次启动自动补发验证 |
| **自动补发** | 启动后自动重试验证 | 用户无感知，自动开通权限 |

#### 清理模式是否应该在生产环境？

**✅ 绝对应该！这是核心保护机制！**

**真实用户场景**：

1. **用户支付时应用崩溃** (最常见)
   ```
   用户: "我付了钱，为什么没有开通？"
   
   没有清理模式:
     ❌ 交易卡在 pending
     ❌ 用户无法再次购买
     ❌ 需要联系客服处理
   
   有清理模式:
     ✅ 下次启动自动处理
     ✅ 自动验证并开通
     ✅ 用户无感知
   ```

2. **网络不稳定**
   ```
   用户在地铁/电梯中购买
   支付成功，但网络断开
   验证失败
   
   没有清理模式:
     ❌ 收据丢失
     ❌ 无法补发
     ❌ 用户权益受损
   
   有清理模式:
     ✅ 收据已保存
     ✅ 网络恢复后自动补发
     ✅ 权限自动开通
   ```

3. **系统资源不足**
   ```
   低端设备/内存不足
   应用被系统杀死
   
   没有清理模式:
     ❌ 交易状态未知
     ❌ 需要手动处理
   
   有清理模式:
     ✅ 自动恢复状态
     ✅ 自动完成交易
   ```

### 2. 超时机制分析

#### 当前的超时设置

| 阶段 | 超时时间 | 位置 | 是否足够 |
|------|---------|------|---------|
| **清理模式** | 3秒 | `_clearPendingTransactions` | ✅ 合理 |
| **手动清理** | 每次3秒，共3次 | `clearAllPendingTransactions` | ✅ 合理 |
| **服务器验证** | ❌ 无超时 | `_verifyReceiptWithServer` | ⚠️ 需要添加 |
| **UI Loading** | ❌ 无超时 | `confirm_payment_page.dart` | ⚠️ 需要添加 |

#### 问题所在：UI 层面无超时

**当前代码**：
```dart
// confirm_payment_page.dart
EasyLoading.show(status: '正在支付...');

// 1.5秒后更新提示
Future.delayed(const Duration(milliseconds: 1500), () {
  if (EasyLoading.isShow) {
    EasyLoading.show(status: '正在验证...');  // ← 一直显示，无超时！
  }
});

// iOS 内购回调
callback: (success, errorMessage) {
  EasyLoading.dismiss();  // ← 只有回调时才会关闭
}
```

**问题**：
- 如果回调丢失（StoreKit Bug）
- UI 会一直显示"正在验证..."
- 用户看起来卡住了
- 但实际上清理模式已经超时完成（3秒）

### 3. 建议的改进方案

#### 改进 1: 添加 UI 超时保护

```dart
// 添加超时定时器
Timer? _iapTimeout;

Future<void> _handlePayment() async {
  try {
    EasyLoading.show(status: '正在支付...');
    
    // ✅ 设置 30 秒超时
    _iapTimeout = Timer(const Duration(seconds: 30), () {
      if (EasyLoading.isShow) {
        EasyLoading.dismiss();
        EasyLoading.showError('支付超时，请稍后重试');
      }
    });
    
    // 内购操作
    await _paymentService.pay(/* ... */);
    
  } catch (e) {
    _iapTimeout?.cancel();
    EasyLoading.dismiss();
  }
}

// 回调中取消超时
callback: (success, errorMessage) {
  _iapTimeout?.cancel();  // ✅ 取消超时
  EasyLoading.dismiss();
  
  if (success) {
    // 成功处理
  } else {
    EasyLoading.showError(errorMessage ?? '支付失败');
  }
}
```

#### 改进 2: 添加服务器验证超时

```dart
Future<Map<String, dynamic>> _verifyReceiptWithServer({
  required String receiptData,
  required String orderId,
  // ...
}) async {
  try {
    final response = await _dioClient.post(
      '/c/pay/applepay',
      data: { /* ... */ },
      options: Options(
        contentType: Headers.jsonContentType,
        validateStatus: (status) => true,
        // ✅ 添加超时
        sendTimeout: const Duration(seconds: 30),
        receiveTimeout: const Duration(seconds: 30),
      ),
    );
    
    // ...
  } on DioException catch (e) {
    if (e.type == DioExceptionType.sendTimeout ||
        e.type == DioExceptionType.receiveTimeout) {
      // 超时处理
      return {
        'success': false,
        'message': '验证超时，请稍后重试',
      };
    }
    throw NetworkException('网络请求失败: ${e.message}');
  }
}
```

## 📊 为什么这次会卡在"正在验证"？

### 日志分析

```
1009: ⏳ 等待支付结果...
1010: 💡 提示已更新：正在验证收据
1011: 清理超时，可能没有待处理交易    ← 清理模式已完成（3秒）
1012: 清理模式：完成
...
1018: 购买失败: storekit_duplicate_product_object
```

**时间线**：

```
00:00  用户点击购买
       ↓
       UI 显示："正在支付..."
       ↓
00:00  开始清理模式
       ↓
00:03  清理模式超时完成  ← 日志1011
       ↓
       继续购买流程
       ↓
       弹出支付确认框
       ↓
       用户输入密码
       ↓
       支付成功（可能）
       ↓
00:05  购买失败报错: duplicate_product_object  ← 日志1018
       ↓
       触发 _handlePendingTransaction
       ↓
00:05  第1次尝试恢复购买
       ↓
00:08  第2次尝试恢复购买
       ↓
00:11  处理完成（但无回调）
       ↓
       回调通知 UI
       ↓
00:11  UI 关闭 Loading ✅
```

**所以你看到的"正在验证"卡住**：
- 实际时长：约 11 秒（清理3秒 + 处理8秒）
- UI 没有超时机制
- 如果 _handlePendingTransaction 也失败
- 用户会一直看到"正在验证"
