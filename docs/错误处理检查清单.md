# 错误处理检查清单

## 🎯 目的

确保所有 Provider 层正确处理错误，不向用户显示技术性错误信息。

---

## ✅ 快速检查命令

在项目根目录运行以下命令，快速定位可能有问题的代码：

### 1. 检查是否使用了 `e.toString()`（❌ 错误）

```bash
cd /Users/mac/Desktop/vueToFlutter/yakaixin_app

# 查找所有在 catch 块中使用 e.toString() 的 Provider
grep -rn "e.toString()" lib/features/*/providers/*.dart | grep -v "e.error"
```

**预期结果**: 应该返回空（没有匹配）

**如果有匹配**: 需要修改为 `e.error?.toString() ?? '默认提示'`

---

### 2. 检查是否正确捕获了 DioException

```bash
# 查找所有 catch (e) 但没有指定类型的情况
grep -rn "} catch (e)" lib/features/*/providers/*.dart
```

**预期结果**: 应该有两种情况：
- `on DioException catch (e)` ✅
- 最后的兜底 `catch (e)` ✅

**如果发现**: 单独的 `catch (e) { ... e.toString() ...}` ❌ 需要改为 `on DioException catch (e)`

---

### 3. 检查是否使用了 `e.message`（❌ 错误）

```bash
# 查找使用 e.message 的地方
grep -rn "e.message" lib/features/*/providers/*.dart
```

**预期结果**: 应该返回空（DioException.message 可能为 null）

**如果有匹配**: 需要修改为 `e.error?.toString() ?? '默认提示'`

---

### 4. 检查 EasyLoading/Toast 错误显示

```bash
# 查找所有使用 EasyLoading.showError 的地方
grep -rn "EasyLoading.showError" lib/features --include="*.dart"
```

**检查每个匹配**：
- ✅ 确保参数不是 `e.toString()`
- ✅ 确保参数是 `errorMsg` 或 `e.error?.toString() ?? '默认提示'`

---

### 5. 检查是否有未处理的异常

```bash
# 查找所有 async 函数但没有 try-catch 的情况（简化检查）
grep -rn "Future<void>" lib/features/*/providers/*.dart | head -20
```

**手动检查**: 每个异步方法是否都有 try-catch 包裹

## 📋 标准模式检查清单

### ✅ 正确的 Provider 错误处理

```dart
Future<void> someAction() async {
  try {
    state = state.copyWith(isLoading: true, error: null);
    
    // 调用 Service
    final result = await _service.getData();
    
    state = state.copyWith(data: result, isLoading: false);
    
  } on DioException catch (e) {
    // ✅ DioException: 使用拦截器已处理的友好信息
    final errorMsg = e.error?.toString() ?? '操作失败，请稍后重试';
    state = state.copyWith(
      isLoading: false,
      error: errorMsg,
    );
    
  } catch (e) {
    // ✅ 其他异常: 使用兜底提示
    state = state.copyWith(
      isLoading: false,
      error: '操作失败，请稍后重试',
    );
  }
}
```

### ❌ 错误示例1：直接使用 e.toString()

```dart
} catch (e) {
  state = state.copyWith(error: e.toString());  // ❌ 会显示技术错误
}
```

### ❌ 错误示例2：使用 e.message

```dart
} on DioException catch (e) {
  state = state.copyWith(error: e.message);  // ❌ e.message 可能为 null
}
```

### ❌ 错误示例3：不区分 DioException 和其他异常

```dart
} catch (e) {
  final errorMsg = e.toString();  // ❌ 无法获取拦截器处理的友好信息
  state = state.copyWith(error: errorMsg);
}
```

## 🔍 逐文件检查清单

### Provider 文件检查表

对于每个 `lib/features/*/providers/*.dart` 文件：

- [ ] ✅ 所有 `async` 方法都有 `try-catch`
- [ ] ✅ 捕获异常时使用 `on DioException catch (e)`
- [ ] ✅ 获取错误信息使用 `e.error?.toString() ?? '默认提示'`
- [ ] ✅ 不使用 `e.toString()` 或 `e.message`
- [ ] ✅ 有兜底的 `catch (e)` 处理未预期错误
- [ ] ✅ 兜底 catch 提供固定的友好提示

### View 文件检查表

对于每个 `lib/features/*/views/*.dart` 文件：

- [ ] ✅ 使用 `ref.listen()` 监听 `state.error` 并显示 Toast
- [ ] ✅ 或在 Provider 中直接使用 `EasyLoading.showError(errorMsg)`
- [ ] ✅ 不要同时在 Provider 和 View 中显示错误（避免重复）
- [ ] ✅ 不直接显示 `e.toString()`

## 📝 总结

### 核心要点

1. **Provider 层**:
   - 必须使用 `on DioException catch (e)`
   - 必须使用 `e.error?.toString() ?? '默认提示'`
   - 必须有兜底 `catch (e)` 提供固定友好提示

2. **不要使用**:
   - ❌ `e.toString()`
   - ❌ `e.message`
   - ❌ 单独的 `catch (e) { error: e.toString() }`

3. **验证**:
   - 运行所有检查命令
   - 手动测试各种错误场景
   - 确保用户永远看不到技术错误
