# 视频播放页「目录数据」与小程序对照分析

## 一、小程序逻辑（newVideo + study/detail）

### 1. 进入视频页时的参数（study/detail index.vue Line 814）

```text
pages/study/newVideo/index?
  url=...&
  filter_goods_id=${courseItem.id || ""}   ← 关键：用 courseItem.id（套餐商品 id）
  &class_id=${courseItem.class_id}
  &order_id=...&goods_id=...&goods_pid=...&lesson_id=...&system_id=...&name=...&status=...
```

- `courseItem` 来自 **learnCourseList**，数据源接口：`study.courseDetailLessons` → **GET /c/study/learning/series/goods**
- 每条 course 项有：**id**（套餐商品 id）、**class_id**、name、lesson 等
- 目录用 **filter_goods_id = courseItem.id**，与 getGoodsDetail 返回的 **detail_package_goods[].id** 一致

### 2. newVideo 内：getID()（Line 370-383）

从 pageData（即上述 query）取值：

- goods_id、**filter_goods_id**、goods_pid、order_id、url、system_id、name、class_id、status

### 3. getDetail()（Line 386-396）

- 接口：**GET /c/goods/v2/detail**
- 参数：goods_id, **user_id**(student_id), **student_id**, **merchant_id**, **brand_id**, **no_professional_id: '1'**

### 4. 选 currentGood（Line 407-415）

- 若有 **filter_goods_id** → currentGood = filter_goods_id
- 否则 → currentGood = res.data.detail_package_goods[0].id

### 5. getChapter()（Line 325-332）

- 接口：**GET /c/goods/v2/chapter**
- 参数：**goods_id: this.currentGood**, **student_id**, **no_professional_id: "1"**

## 二、Flutter 当前逻辑

### 1. 进入视频页时的参数（course_detail_page）

- 传的是：**filterGoodsId: classItem.classId**、classIndex、goodsId、orderId、systemId 等
- classItem 来自 **classList**，数据源：**GET /c/study/learning/series/goods**（与小程序同一接口）
- **CourseClassModel 只解析了 class_id、name、lesson 等，没有解析 "id"**
- 因此 Flutter 只能用 **classId** 当 filterGoodsId，而小程序用的是 **courseItem.id**

### 2. 视频页 initState

- _goodsId = widget.goodsId  
- _filterGoodsId = widget.filterGoodsId（当前即 classId）  
- _classIndex = widget.classIndex  

### 3. getGoodsDetail（video_service）

- 接口：GET /c/goods/v2/detail  
- 参数：goods_id, user_id, student_id, **merchant_id, brand_id, no_professional_id: '1'**  
- **与小程序一致**

### 4. 选 actualGoodsId（video_index_page）

- 优先 **classIndex** → detail_package_goods[classIndex].id  
- 否则：若 filterGoodsId 在 packageIds 中则用 filterGoodsId，否则用 detail_package_goods[0].id  
- 问题：filterGoodsId 传的是 **classId**，往往不在 detail_package_goods[].id 列表中（id 与 class_id 可能不同），所以实际常走 classIndex 或第一个套餐，若 **series/goods 与 getGoodsDetail 的 detail_package_goods 顺序不一致**，目录就会错。

### 5. getChapterList（video_service）

- 接口：GET /c/goods/v2/chapter  
- 参数：goods_id, student_id, no_professional_id: '1'  
- **与小程序一致**

## 三、差异总结

| 项目           | 小程序                         | Flutter（修复前）                    |
|----------------|--------------------------------|--------------------------------------|
| 接口           | /c/goods/v2/detail、/c/goods/v2/chapter | 相同                                 |
| getGoodsDetail 参数 | goods_id, user_id, student_id, merchant_id, brand_id, no_professional_id | 已对齐                               |
| chapter 参数   | goods_id, student_id, no_professional_id | 已对齐                               |
| 进入视频页传参 | **filter_goods_id = courseItem.id** | **filterGoodsId = classItem.classId** |
| 选目录用 id    | currentGood = filter_goods_id（即 courseItem.id） | actualGoodsId 来自 classIndex 或 classId，**无 courseItem.id** |

**根因**：同一接口 `/c/study/learning/series/goods` 返回的每条班级有 **id**（套餐商品 id）和 **class_id**。小程序用 **id** 作为 filter_goods_id 拉目录；Flutter 未解析 **id**，只传了 **classId**，且用 classIndex 依赖「列表顺序一致」，一旦顺序不一致或 class_id ≠ detail_package_goods[].id，目录就会错。

---

## 四、修复方案

1. **CourseClassModel 增加字段 id**（对应 JSON "id"），与小程序 courseItem.id 一致。  
2. **课程详情跳转录播时**：传 **filterGoodsId: classItem.id ?? classItem.classId**。  
3. 视频页逻辑保持不变：优先用 filterGoodsId 在 detail_package_goods 的 id 列表中则用，否则再 classIndex / 第一个套餐。  
4. 运行 **build_runner** 重新生成 freezed/json 代码。

这样 Flutter 与小程序在「用哪个 id 拉目录」上完全一致，目录数据即可一致。

---

## 五、接口确认

**目录接口与小程序一致**：

- 路径：`/c/goods/v2/chapter`（请求时 = baseUrl + path = **/api/c/goods/v2/chapter**）
- 小程序：`chapterpaper` → GET，url: `/c/goods/v2/chapter`，data: goods_id, student_id, no_professional_id
- Flutter：`getChapterList` → GET `/c/goods/v2/chapter`，queryParameters: goods_id, student_id, no_professional_id: '1'

选目录用哪个 goods_id 的优先级已与小程序对齐：**有 filter_goods_id 且在 detail_package_goods 的 id 列表中则用 filter_goods_id**，否则再兜底 classIndex 或第一个套餐。

## 六、为什么有时传错 goods_id（589882702371491422 vs 593505082385898224）

- **593505082385898224**：章节接口返回数据里每条章节的 `goods_id`，即**套餐商品 id**，传它时目录正确。
- **589882702371491422**：来自**课程详情页的 widget.goodsId**（主商品 id，进入课程详情时的路由参数）。

传递链：

1. 课程详情页用 **widget.goodsId**（主商品 id，如 589882702371491422）调接口、并作为 **goodsId** 传给视频页。
2. **filterGoodsId** = classItem.id ?? classItem.classId。若 `/c/study/learning/series/goods` 里某班级的 **id** 等于主商品 id，或未返回 **id** 只用了 **class_id**，则 filterGoodsId 也可能是 589882702371491422。
3. 视频页 getGoodsDetail(主商品 id) 返回的 **detail_package_goods** 里，可能**第一个元素的 id 也是主商品 id**，所以 packageIds 里包含 589882702371491422。
4. 当前逻辑：filterGoodsId 在 packageIds 中则用 filterGoodsId → 就会用 589882702371491422 调 chapter，而章节接口需要的是**套餐 id**（593505082385898224），导致目录错。

修复：**当 filterGoodsId 与主商品 id（_goodsId）相同时，不用于 chapter**，改为用 classIndex 或 detail_package_goods[0] 得到的**套餐 id**，保证传给 `/c/goods/v2/chapter` 的始终是套餐 id（如 593505082385898224）。
