# 移除支付超时机制

**更新时间**: 2025-01-25  
**变更类型**: 移除功能

---

## 📝 变更说明

### 背景

之前为了防止 iOS 内购回调丢失导致 UI 无限等待，添加了 30 秒超时机制。但根据实际测试，这个超时可能在正常支付流程中被触发，影响用户体验。

### 变更内容

**移除内容**：
- ❌ 30 秒超时 Timer
- ❌ 超时标志位 `isTimeout`
- ❌ 超时警告日志
- ❌ 超时错误提示

**保留内容**：
- ✅ Completer 异步等待机制
- ✅ 支付回调处理
- ✅ EasyLoading 状态提示

## 🔄 修改前后对比

### 修改前

```dart
// ❌ 有超时机制
final completer = Completer<wechat_pay.PaymentResult>();
bool isPurchaseCompleted = false;

// 30 秒超时
bool isTimeout = false;
Timer? timeoutTimer;
timeoutTimer = Timer(const Duration(seconds: 30), () {
  if (!completer.isCompleted && !isPurchaseCompleted) {
    isTimeout = true;
    debugPrint('⚠️ 支付超时（30秒无回调）');
    
    completer.complete(
      wechat_pay.PaymentResult.failed(
        '支付超时\n\n如已完成支付，请重启应用查看权限'
      )
    );
  }
});

await paymentService.pay(
  // ...
  onResult: (success, errorMessage) {
    if (!completer.isCompleted && !isTimeout) {
      timeoutTimer?.cancel();  // 取消定时器
      isPurchaseCompleted = true;
      // ...
    }
  },
);

print('   ⏳ 等待支付结果（30秒超时）...');
final result = await completer.future;
```

### 修改后

```dart
// ✅ 无超时机制，纯等待回调
final completer = Completer<wechat_pay.PaymentResult>();

await paymentService.pay(
  // ...
  onResult: (success, errorMessage) {
    if (!completer.isCompleted) {
      print('\n👉 收到内购结果回调');
      print('   成功: $success');
      print('   错误: $errorMessage');
      
      if (success) {
        completer.complete(wechat_pay.PaymentResult.succeeded());
      } else {
        completer.complete(wechat_pay.PaymentResult.failed(errorMessage ?? '内购失败'));
      }
    }
  },
);

// ✅ 无限期等待（直到收到回调）
print('   ⏳ 等待支付结果...');
final result = await completer.future;
```

## 📊 影响分析

### 用户体验

| 场景 | 修改前 | 修改后 | 改善 |
|------|--------|--------|------|
| **正常支付** | 可能误触发超时 | ✅ 正常完成 | ✅ 更好 |
| **慢速网络** | 30秒后强制失败 | ✅ 等待完成 | ✅ 更好 |
| **用户取消** | 无限等待 | ⚠️ 无限等待 | ⚠️ 相同 |
| **回调丢失** | 30秒后提示 | ⚠️ 无限等待 | ❌ 更差 |

### 技术影响

**优点**：
- ✅ 避免误触发超时
- ✅ 支持慢速网络环境
- ✅ 代码更简洁

**缺点**：
- ❌ 回调丢失时无超时保护
- ❌ 用户取消时可能需要强制退出

## 🛡️ 现有保护机制

### 1. 收据缓存机制

```dart
// ✅ 支付成功后立即缓存收据
await IAPReceiptCache.saveFailedReceipt(
  orderId: _currentOrderId!,
  receiptData: receiptData,
  // ...
);
```

**作用**：
- 即使 App 崩溃，收据也不会丢失
- 下次启动时自动验证

### 2. 自动补发机制

```dart
// ✅ 应用启动时自动补发
Future<bool> initialize() async {
  await _clearPendingTransactions();
  await retryFailedReceipts();  // ← 自动补发
  return true;
}
```

**作用**：
- 自动验证缓存的收据
- 补发权限

### 3. 清理模式

```dart
// ✅ 清理 pending transactions
Future<void> _clearPendingTransactions() async {
  _isCleaningMode = true;
  await _iap.restorePurchases();
  // ...
}
```

**作用**：
- 清理未完成的交易
- 避免重复购买错误

## 🎯 为什么移除超时？

### 真实场景分析

**场景 1: 沙盒环境测试**
```
问题：
- 沙盒环境网络可能很慢
- 回调延迟 10-20 秒很常见
- 30 秒超时可能太短

结果：
- 用户已完成支付
- 但超时机制误判为失败
- 造成困惑
```

**场景 2: 正常支付流程**
```
流程：
1. 点击支付 (0秒)
2. 输入密码 (0-30秒)  ← 用户可能考虑很久
3. 确认支付 (30秒)
4. StoreKit 处理 (30-35秒)
5. 触发回调 (35秒)

结果：
- 如果步骤2用时较长
- 可能在步骤5触发超时
- 实际上支付已成功
```

### StoreKit 回调特性

**Apple 官方说明**：
```
StoreKit 回调时机不确定：
- 通常几秒内返回
- 但可能延迟到几分钟
- 不应该设置固定超时

推荐做法：
✅ 使用 purchaseStream 监听
✅ 使用收据缓存防止丢失
✅ 提供恢复购买功能
❌ 不要设置固定超时
```

## ✅ 总结

### 变更理由

1. **避免误判**：30秒超时可能在正常流程中触发
2. **符合Apple最佳实践**：不应设置固定超时
3. **已有保护机制**：收据缓存 + 自动补发 + 清理模式

### 现有保护

| 机制 | 作用 | 防止问题 |
|------|------|---------|
| **收据缓存** | 支付后立即保存 | 防止数据丢失 |
| **自动补发** | 启动时验证 | 补发权限 |
| **清理模式** | 清理pending | 避免重复购买 |
| **用户指引** | 提示重启 | 用户自救 |

### 最终方案

**✅ 移除超时，依赖现有保护机制**
- 更符合 StoreKit 特性
- 避免误判
- 已有完善的补救措施
